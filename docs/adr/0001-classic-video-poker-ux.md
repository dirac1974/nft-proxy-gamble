# ADR 0001 — Classic single-line video poker UX (timing, sound, animation)

- Status: Accepted
- Date: 2026-06-18
- Issue: https://github.com/dirac1974/nft-proxy-gamble/issues/13

## Context

The game is already a provably-fair 9/6 Jacks-or-Better engine, but the `play.tsx`
screen presented it as a generic card UI. We want the look-and-feel of a standard
single-line video-poker cabinet (always-on paytable, LED meters, deal/draw timing,
hold banners, sound) while keeping the existing dark theme and, critically, without
copying any vendor's (e.g. IGT) proprietary assets or trademarks.

## Decision

### Layout
- **Always-on paytable** (9 hands × 5 bet columns) pinned at the top. The current
  bet column is highlighted; the winning row flashes on a paying result. Royal Flush
  shows the classic 4000 jump in the 5-coin column (800× × 5); all other cells are
  per-coin × bet.
- **Cabinet meter bar** (`components/MeterBar.tsx`): three amber LED-style readouts —
  CREDITS, BET, WIN — rendered as a pure presentational component (no animation/state),
  so the rollup logic lives in `play.tsx` and stays unit-testable.
- **Cards**: face-down backs before the deal; deal-in stagger on reveal; a "HELD"
  banner overlaid on held cards (not just a border). Holds are only togglable in the
  `dealt` phase.
- **Controls**: single primary button whose label tracks the phase
  (NEW GAME → DEAL → DRAW → PLAY AGAIN), 1–5 bet chips, an explicit BET MAX (=5), and
  the existing Cash Out (phase `drawn` && balance ≥ 100).

### Timing (ms)
| Event | Behavior | Timing |
|---|---|---|
| Deal | cards reveal left→right (staggered deal-in) | 90 ms stagger/card |
| Hold toggle | border + HELD banner | 200 ms + `hold` sound |
| Draw | only non-held positions re-animate; held cards stay put | 90 ms stagger |
| Win row flash | winning paytable row blinks | 200 ms on / 200 ms off × 3 |
| Win meter rollup | CREDITS/WIN count up, `coinDrop` tick each step | ≤ 18 × 80 ms = 1440 ms (≤ 1500 ms) |
| Win overlay | premium hands only | existing `WinOverlay`, auto-dismiss 2.5 s |

"Premium" = Four of a Kind or better (≥ 25× per coin) → `WinOverlay` + `bigWin` sound.
A normal paying hand → inline result + `win` sound + row flash. A loss → `lose` sound,
no flash, no overlay.

### Sound — original assets only
`soundService.ts` had the load/play plumbing but every asset slot was `null` and it was
never imported. We:
- Added `scripts/gen-sfx.mjs`, a dependency-free Node generator that synthesizes the six
  SFX as mono 22.05 kHz 16-bit PCM WAVs (deal click, hold blip, win triad, bigWin
  arpeggio, lose tone, coin tick) directly from raw samples.
- Wired `SOUND_ASSETS` to `require()` those WAVs (`coinDrop` → `coin.wav`), and call
  `initSounds()` / `unloadSounds()` from `app/_layout.tsx`.
- Call `playSound(...)` from the deal/hold/draw handlers per the timing table.

**These are placeholder originals.** They were procedurally generated by us and contain
no third-party audio. They are intended to be replaced later with a licensed/professional
SFX pack. We deliberately did **not** download or reproduce any IGT (or other vendor)
sounds, card art, or trademarks — we match the *genre conventions* only.

### Animation library
Reuses the existing `react-native-reanimated` setup (already mocked in jest). The
winning-row flash uses `withRepeat(withSequence(...), 3)`; the deal stagger reuses the
`Card` component's `dealIndex` spring. The meter rollup is intentionally JS-driven
(`setInterval`) rather than reanimated, so it is deterministic and testable.

## Consequences
- The UI now reads as a classic cabinet without any proprietary assets.
- Rollup-in-JS keeps the celebratory math unit-testable (a stale-closure bug in the
  rollup target was caught by that test during implementation).
- A literal 3D card flip (back→face) was deferred; the staggered deal-in is the reveal.
  This is a visual refinement that needs a device to judge and can be added later.
- Visual/audio fidelity needs a human's eyes/ears on a device — see `MORNING_REPORT.md`.
